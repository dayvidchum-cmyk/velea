## CRITICAL

**C1 ✅ THE AYANAMSA SPLIT — two of the four Swiss-Ephemeris instances never set Lahiri; they compute in Fagan-Bradley, ~0.883° off the frame the natal charts are stored in.**
`server/vedic/natal-chart-engine.ts` (getSiderealLongitudes/WithSpeed) and `server/profection/transit-calculator.ts` never call `set_sid_mode(LAHIRI)`; only `panchang/astronomy.ts` and `birthchart/calculator.ts` do (verified by grep + the agent verified the 0.883° delta empirically against the WASM lib). Blast radius: every transit-to-natal hit carries a 0.88° bias; sign/nakshatra labels wrong within 0.88° of any boundary; day crown/chandrabala Moon wrong sign ~3% of days; the Arc's 90-day crown/apex scan gets wrong tarabala ~6-7% of scanned days; golden-day and hora-lord dignity flips near cusps; **Varshaphala finds the solar return ~0.9 days late → the whole year chart (lagna/houses) is wrong for every user**; Time Lord ingress dates wrong by 0.883°÷speed (≈22h Sun, ≈9d Jupiter, ≈26d Saturn), and a Time Lord within 0.88° of a cusp shows the wrong sign/house all segment.

**C2 ✅ RAHU IS PLUTO — the Stage-and-guests read queries planet 9/10 as Rahu/Ketu, but in Swiss Ephemeris 9 = Pluto and 10 = Mean Node (= Rahu).**
`server/profection/transit-calculator.ts:145-146` vs `swisseph.js` constants (verified). The premium guests layer reports "Rahu" co-present when *Pluto* shares the sign and "Ketu" co-present exactly when the real *Rahu* is there.

**C3 ✅ systemPrompts IS WIDE OPEN — any logged-in user can rewrite the feature flags and read the tester list.**
`server/routers.ts:1626-1643`: `list` + `upsert` are plain protectedProcedure, no admin check (verified). The `feature_flags` row lives in that table: one crafted `upsert` makes every premium surface free for the whole user base within the 60s flag cache; `list` leaks tester emails; arbitrary prompt keys can be corrupted.

**C4 THE SERVICE WORKER CACHES ITSELF — which silently defeats the v651 stale-instance guard.**
`sw.js:5` `ASSET_RE` matches `/sw.js`; the guard's `fetch("/sw.js", {cache:"no-store"})` only bypasses the HTTP cache, not the SW's cache-first handler. First freshness check stores sw.js; every later check reads that copy → `server === mine` forever → the reload never fires for exactly the resumed-stale-PWA case it was built for.
